# 医院多类型报修工单系统设计方案

> 适用场景：水电维修、物业维修、电梯维修等多类型报修，各病区扫码报修，流转至各班组负责人，派单、处理、完工上传照片全流程闭环。

---

## 一、系统角色与流程

```
病区扫码（报修人）
    ↓ 提交报修单（含照片、位置、类型）
系统自动分流（按维修类型）
    ├─ 水电类 → 水电组负责人
    ├─ 物业类 → 物业组负责人
    └─ 电梯类 → 电梯组负责人
            ↓ 派单给具体维修工
        维修工接单 → 处理 → 上传完工照片
            ↓
        报修人确认 → 关闭工单
            ↓
        后台统计（响应时长、维修类型分布、各班组工作量）
```

**核心要素**：
- 4 类角色：报修人、组长、维修工、管理员
- 5 种状态：待派单 / 已派单 / 处理中 / 待确认 / 已完成
- 多种维修类型（可扩展）
- 图片上传（报修现场 + 完工）
- 全流程留痕（谁、什么时候、做了什么）

---

## 二、技术方案架构

```
┌──────────────────────────────────────────┐
│ 病区墙上贴 QR Code（每个病区一个,带位置编码）│
└────────────┬─────────────────────────────┘
             ↓ 扫码
┌──────────────────────────────────────────┐
│ GitHub Pages 静态网页                     │
│ ├ 报修页（护士填表+拍照）                   │
│ ├ 工单大厅（组长看自己组的单）               │
│ ├ 我的工单（维修工接单）                    │
│ └ 管理后台（你看全部+统计）                 │
└────────────┬─────────────────────────────┘
             ↓ API 调用
┌──────────────────────────────────────────┐
│ Supabase                                  │
│ ├ 数据库（工单、人员、班组）                 │
│ ├ Storage（报修照片、完工照片）             │
│ ├ Auth（登录权限）                         │
│ └ Realtime（新工单实时推送给组长）           │
└──────────────────────────────────────────┘
```

**总成本：0 元**（全部免费额度，日工单量 < 500 单完全够用）

## 三、数据库设计（Supabase）

## 四、四个功能网页（GitHub Pages）

### 网页 1：报修页 `/report.html?ward=W3F2`
- 扫码自动带入病区
- 选维修类型（图标按钮）
- 文字描述故障
- 直接调相机拍照（手机浏览器原生支持）
- 留报修人姓名 + 分机
- 提交 → 存 Supabase

### 网页 2：组长大厅 `/leader.html`
- 登录后只看本组待派单工单
- 实时推送（有新单弹通知）
- 一键派给组员
- 看本组工作量统计

### 网页 3：维修工页 `/worker.html`
- 看派给我的单
- 点"开始处理"→ 状态变更
- 完工拍照上传
- 一键完工

### 网页 4：管理后台 `/admin.html`（管理员用）
- 全院工单总览
- 各类报修分布饼图
- 平均响应时长趋势
- 各班组工作量排行
- 异常工单预警（超 24h 未处理）
- 月报一键导出

## 五、关键设计要点（避坑）

### 1. 病区 QR Code
- 每个病区一个独立 QR Code，URL 带病区编码
- 例：`https://yourname.github.io/repair/?ward=W3F2`（三楼东病区）
- 护士扫码后不用选病区，直接填故障

### 2. 报修页不要求登录
- 报修页完全开放，只填姓名 + 分机（像问卷星）
- 维修工和组长才需要登录（数量少、可控）

### 3. 紧急工单逃生通道
- "特急"按钮 → 立即打电话 + 短信通知组长
- 别让紧急情况（漏水、停电、电梯困人）卡在系统里

### 4. 照片用 Supabase Storage
- 不要存 GitHub（仓库会膨胀）
- Supabase Storage 免费 1GB，每张压缩到 200KB 可存 5000 张
- 数据库只存 URL

### 5. 数据安全
- 网页代码可公开，数据靠 Supabase 权限（RLS）保护
- 前端只用 anon key，通过 RLS 限制每人只看该看的数据

### 6. 离线兜底
- 保留电话报修通道作为备份
- 第一年并行运行，半年后再完全切换

## 六、落地路径（分阶段）

### 第一阶段（1-2 周）：MVP 最小可用版
- 只做 1 个网页：报修页 + 一张工单表
- 不分班组，所有工单先到管理员手动派单（微信群通知）
- 目标：验证护士愿不愿意用扫码报修

### 第二阶段（1 个月）：加分流
- 加组长账号、班组自动分流
- 加维修工接单页
- 实时通知

### 第三阶段（2-3 个月）：完整闭环
- 完工确认
- 统计后台
- 月报自动化

### 第四阶段（6 个月）：优化
- 高频故障分析 → 预防性维护建议
- 维修工 KPI 报表
- 接入医院 OA 单点登录（如果可能）

## 八、合规与立项建议

1. **先报信息科**：确认是否允许使用 Supabase（海外服务器）；如不允许，换国内服务器 + 相同架构
2. **先做内部试点**：1-2 个病区跑半年，脱敏数据，作为科研项目立项
3. **避开患者数据**：工单只涉及设备和位置，不带任何患者信息
4. **数据备份**：Supabase 数据每月导出一份存本地
