## Q1：frontmatter 格式与添加方式

### 基本格式
用 `---` 包夹的 YAML 块，置于文件最顶端：

```yaml
---
title: "標題"
date: "2026-04-06"
tags:
  - 標籤A
  - 標籤B
---
```

- `key: value` 形式
- 多值用 YAML 列表
- 字符串可加引号

### CLAUDE.md 最低要求
所有新建笔记**必须含** `title`、`date`、`tags`。

### 三类模板字段

| 模板 | 关键字段 |
|---|---|
| 每日筆記 | date / week / day / type / tags |
| 週計畫 | week / date_start / date_end / type / tags |
| 知識庫頁面 | title / type / source / created / tags / related |

### 添加方式
1. **手动**：直接写 `---...---`
2. **模板**：放 `Templates/`，Templater 展开 `<% tp.date.now() %>`
3. **Claude 代写**：说「幫我新增到筆記」，按规则归类

## Q2：参考文献、文档、日记的 frontmatter 一样吗？

**不一样。**三者归不同资料夹。

### 对照

| 项目 | 参考文献 | 文档（知识库） | 日记 |
|---|---|---|---|
| 资料夹 | `Clippings/` | `知識庫/` 或 `創作庫/` | `每日筆記/` |
| `type` | clipping | 知識庫 | 每日筆記 |
| 来源 | `source: URL` | `source: 來源` | 无 |
| 时间 | date / clipped | created | date + week + day |
| 是否可改 | 不要修改 | AI 写、你审 | 自己写 |

### 三种范例

**参考文献**
```yaml
title: "原文標題"
source: "https://..."
date: 2026-05-01
tags: [clipping]
```

**文档（知识库）**
```yaml
title: ""
type: 知識庫
source: ""
created: 2026-05-01
tags: []
related: []
```

**日记**
```yaml
date: 2026-05-01
week: 2026-W18
day: 星期五
type: 每日筆記
tags: [每日筆記]
```

### 差异要点
- `type` 决定归类与是否被消化
- 只有前两类需要 `source`
- 日记多 `week` / `day` 用于周报

## Q3：参考文献加入 Clippings 后的全流程

### 流程图

```
网页文章
   │ Web Clipper（10 秒）
   ▼
① Clippings/ 原始输入（不动它）
   │ 周日 09:17 自动 / 手动「消化 Clippings」
   ▼
② 知识重整七步骤
   ▼
③ 知識庫/xxx.md 结构化主题页
   │ [[wiki link]] 织网
   ▼
④ 創作庫/ 教案、教材引用
   ▼
⑤ 创作回流补强知识库
```

### 各阶段
1. **收集**：Web Clipper → Clippings/，只进不改
2. **触发消化**：周日 cron `17 9 * * 0`，或手动「消化 Clippings」
3. **跨篇消化**：多篇相关文章合并成**主题群页面**
4. **lint**：自动加 wiki link、找缺口、标孤儿
5. **入册**：更新 index.md、log.md
6. **周报**：`每日筆記/YYYY-WXX 知識重整.md`
7. **长期发酵**：被搜索、被引用、被新内容回流

### 一句话
剪 → 囤 → 周日消化 → 主题合并 → 织网入册 → 长期回流。价值是**异步、累积、网络式**。

## Q4：写一篇能耗平台论文时第二大脑如何工作

### 流程

```
你：「幫我寫一篇能耗平台論文」
   ▼
① 跨层检索（Clippings + 知識庫 + 創作庫）
   ▼
② 弹药够不够？
   ├─ 够 → 起草
   └─ 不够 → 反向提示"知识缺口"
   ▼
③ 創作庫/能耗平台論文.md 起草
   引用 [[知识库页]] 与 Clippings 原始来源
   ▼
④ 创作回流：新概念反哺知識庫
```

### 三层各自角色

| 层 | 角色 |
|---|---|
| Clippings/ | 原始文献（引用来源） |
| 知識庫/ | 结构化论据（直接成段） |
| 創作庫/ | 你过去的观点（保持一致性） |

### 关键机制
- **知识缺口反推**：缺料时反向告诉你该去找什么
- **可追溯**：每段论点 `[[wiki link]]` 回连原文
- **回流增强**：新提炼概念自动入知识库

### 一句话
说一句话 → 三层调度 → 缺料反推 → 起草全程回连 → 新概念反哺，越用越厚。

## Q5：修改论文方式与回流时机

### 一、修改 = 改原文件
- 一份论文一个文件
- 第三步原文："读取創作庫/ 中本週**新增或修改**的笔记"
- 例外：说「重寫一版」时可能新建副本

### 二、回流两种触发

**A. 被动（默认，自动）**
- 周日 09:17 cron 跑七步骤
- 第三步批量回流整周改动
- 周报通知

**B. 主动（手动，立即）**
- 「跑一次知識重整」→ 全套七步
- 「幫我把這篇論文的新概念回流到知識庫」→ 单跑第三步

### 三、何时用哪种

| 改动类型 | 推荐方式 |
|---|---|
| 改错字、调语序 | 等周日自动 |
| 新章节、新概念、新模型 | 立刻手动 |
| 成果要用到下一篇 | 必须先手动回流 |

### 四、回流是提炼不是搬运
只抽取**可复用的新概念**进知识库，不复制整段论文。小修改"无事发生"是正常的。

### 一句话
**改原文件**；默认周日自动回流，**有重大概念产出时主动触发**立即回流。
